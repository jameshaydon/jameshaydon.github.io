What even is agentic programming?

14 minute read Published: 2026-02-08

"Agents", "agentic AI" are all the rage right now, but surprisingly little clarity about what it means from a programming perspective. In this post I want to survey current agentic architectures and patterns, and then try to pin it down as a programming concept. Then I'll speculate on how programming languages will evolve to fit this new paradigm.

There is a spectrum:

  • Who is in control, the program or the LLM?
  • How much interaction is there between the LLM and the program?
  • How much interaction is there between the LLM and program with user(s)?

What's been most explored so far are the extremes:

  • Programs which remain very much in control, but make calls to LLMs to power some features. For example papersort will automatically sort lots of documents into directories for you, it makes significant use of an LLM, but the program is in control.
  • An LLM is the main control flow, possibly with significant access to tools. Claude Code is in this category.

What I'm calling "agentic programming", is the very interesting middle-ground:

An agentic program is a program in which LLM calls affect the control flow in some significant manner.

In an agentic program, the LLM's output steers the program: choosing which branch to take, which tool to call, whether to loop or halt, what question to ask next. The LLM participates in the logic of the program, not just its data.

The Basic LLM API

At bottom, the API that LLM providers give you is simple. You send a sequence of messages (a conversation history) and you get back a completion. The completion is either text and/or some structured tool calls: a request to invoke some named function with some arguments.

complete : ([Message], [Tool]) → (Text, [Call])

where Tool specifies an available tool/function on the client (mostly just the JSON schema of it's input/output), and Call is the tool calls the LLM has decided to execute.

The Tools can be thought of the docs of some functions made available to the LLM:

-- Get the current location of the robot
current_location :: () -> Pos
-- Move the robot
move :: Direction -> Result

and the Calls as applications of these functions to some inputs, which the LLM has judged as useful to execute so that it can see the output, of for the side-effects:

current_location () -- (3, 4)
move Left           -- Success

If the result of a tool call is to be returned back to the LLM, this is done in a message, that is, tool call result form part of the history [Message] sent to the LLM.

There's a bunch of stuff not mentioned here, streaming, indicating if a tool must be called, specifying a maximum number of tokens for the output, etc.

Programmatic Tool Calling (Beta)

Anthropic has released a new feature Programmatic Tool Calling. When this is enabled, instead of making tool calls directly, the LLM writes a small python script that executes in a sandboxed environment on Anthropic's servers. This functions as a coroutine with the client: whenever a tool call function is used, the script is suspended, and the API call is returned to the client with that tool call. When the client makes a new API call (hopefully with the tool call response), the script is resumed.

The API is the same, but this is a profound change. The script can combine the results of tool calls with standard python programming. The tools provided by the client can now be composed programatically. If these LLM API requests are part of some program, these ephemeral python programs are now part of the program.

Some Foreshadowing

Right away let's make the remark that inputs to tool calls must be serialisable, since they have to be specified in JSON by the LLM. This is a significant restriction.

Agent coding patterns

Now let's look at some of the ways people are decided to go about agentic coding. A good guide is Building effective agents by Anthropic, they make the dichotomy between the LLM being in control, which they call agents, and when the program is in control, which they call workflows.

One of the striking things is that all of these concepts have existed before, they are just being rediscovered for agentic programming. First we'll look at workflows, which I think it best understood from the lens of categorical programming.

Workflow pattern: Chaining

Prompt chaining decomposes a task into a sequence of steps, where each LLM call processes the output of the previous one.

The task to be performed can be split up into well defined steps. The full task is the composition of these steps. Some steps might not use an LLM call at all.

Example: Answer a natural language question about a specific database.

The workflow might be:

  • Given an LLM the natural language question, and the SQL schema, ask for an SQL query.
  • Execute the query.
  • Give an LLM the question and the query result, ask for a summary/insight.

This (categorical) composition. Though note that the composition is not always just composition of functions:

  • it might accumulate a single thread of end-user conversation history.
  • it might have access to some shared global state.
  • it might be streaming: this is particularly useful when composing two LLM steps. The streaming output of one call can feed directly into the next.

Workflow pattern: Routing

Routing classifies an input and directs it to a specialized followup task. This workflow allows for separation of concerns, and building more specialized prompts.

This means that one LLM is asked to produce a value in some sum type (a coproduct). The next step of the workflow after that is a co-cone, which means we are providing a continuation workflow for each variant of the coproduct. In functional programming cocone are realised as pattern matching.

Workflow pattern: Parallelization

LLMs can sometimes work simultaneously on a task and have their outputs aggregated programmatically.

This is cone: a morphism into a product type. In this case several workflows operate on the same input, in parallel, obtaining independent results. These results are aggregated into a product type (a record), for further processing, by the program, or more agentic workflows.

Workflow pattern: Orchestrator-workers

In this pattern, a main LLM, the orchestrator, will define tasks which sub-agents work on. A synthesizer (which might be the orchestrator or not) will process the results. This is a common pattern used in Claude Code.

Agents: ReAct loop

The ReAct pattern puts the LLM squarely in control of some task. It is free to make tool calls, the results are added to the message history and the LLM is re-invoked. There is no "program" (apart from the code in the tools), the LLM is the control flow.

This is total inversion of control (IoC), the program isn't calling the LLM, the LLM is calling the programs provided to it.

This pattern has already been deployed to great success, but it is limited.

Human in the Loop

Many agentic programs involve a human interactant — a chat interface where a person provides instructions, answers questions, or approves actions. This is "human in the loop."

From a control flow perspective, the human is another source of nondeterminism, another entity that can steer the program. The program now alternates between three parties: itself, the LLM, and the human. The human might:

  • Provide the initial goal.
  • Answer clarifying questions from the LLM.
  • Approve or reject proposed actions.
  • Redirect the agent mid-task.

Here's a question that I think deserves more attention: do humans and LLMs have the same interface?

At the API level, they're both producing messages that get appended to a conversation. Both can, in principle, issue instructions or answer questions. But in practice, their roles are usually asymmetric. The human is typically cast as the "user" — the one with authority and intent — while the LLM is the "assistant" — the one that serves and executes. This asymmetry is baked into the API (the role field in messages) and into the system prompt conventions.

But it doesn't have to be this way. You could imagine architectures where the human and the LLM have symmetric interfaces, where either one can call tools or delegate to the other. In multi-agent systems, you often see LLMs playing different roles in the same conversation. The "user" and "assistant" distinction starts to look like an implementation detail rather than a fundamental boundary.

I think there's interesting PLT work to be done here — in designing type systems or protocols that capture the roles and capabilities of different participants in an agentic program, whether they're humans, LLMs, or other services.

Beyond Tool Use: The Code Generation Spectrum

The tool use pattern — LLM chooses from a fixed menu of functions — is the dominant paradigm right now, but it's starting to feel limiting.

The constraint is this: when the LLM can only make one tool call at a time (or even a batch of independent calls), and the orchestrating program determines what happens between calls, you're limited to patterns that the programmer anticipated. The LLM can't say "call A, then if the result contains X, call B with a modified argument, otherwise call C three times in a loop." That kind of compositional, conditional, iterative use of tools requires the LLM to go back and forth through the agent loop, one call at a time, with the orchestrator faithfully executing each step.

This works, but it's slow and brittle. Every round trip has latency. The conversation history grows. And the orchestrator has to be designed to handle arbitrary sequences of tool calls, which means it's essentially an interpreter for an implicit language of tool-call sequences — but a very impoverished one.

The emerging alternative is to let the LLM write code that makes tool calls. Instead of asking the LLM "which tool do you want to call?", you ask it "write a script that accomplishes this task." The script runs in a sandbox, has access to the tools as callable functions, and can use all the standard programming constructs: variables, conditionals, loops, error handling.

This is a significant shift. The LLM is no longer just picking items from a menu — it's programming. And the spectrum between "agent loop with tool calls" and "LLM writes a complete program" is exactly the interesting design space:

Tool callsGenerated code
ControlOrchestrator mediates every stepLLM writes the full plan
CompositionOne (or a batch of) call(s) at a timeArbitrary composition
IterationAgent loop (external)Loops in generated code
FlexibilityFixed tool schemasFull language expressiveness
SafetyEasy to validate each callNeed sandbox/isolation
LatencyRound trip per tool callOne generation, many calls

Most real systems are somewhere in the middle. An agent might use tool calls for simple operations but generate code for complex multi-step tasks. The question is where to draw the line, and what infrastructure you need at each point on the spectrum.

Same Language, Same Environment?

Here's where it gets really interesting from a PL perspective.

When an LLM generates code to run in a sandbox, the generated code and the agentic program that invoked the LLM are typically written in different languages, or at least run in different environments. The agentic program might be Python or TypeScript; the generated code might be Python running in a container, or JavaScript in an isolated VM.

But what if they were the same language, running in the same environment?

Consider: your agentic program is written in Python. It calls an LLM. The LLM generates Python code. Instead of running that code in a sandbox, you exec it in the same process, with access to the same modules, the same state, the same tools. The generated code isn't just calling predefined tools through a schema — it's calling actual functions in your program, composing them in ways you didn't anticipate, maybe even defining new functions that persist for future LLM calls.

This is exciting and terrifying in equal measure.

On the exciting side: the boundary between "the program" and "what the LLM generates" starts to dissolve. The LLM becomes a metaprogramming facility — a way for the program to extend itself at runtime. You get a kind of dynamic self-modification that would be very hard to achieve otherwise. The LLM can inspect the program's own code, generate new components, and wire them together.

On the terrifying side: you've just given an LLM arbitrary code execution in your process. The safety implications are obvious. But beyond safety, there are deep questions about semantics and reasoning:

  • Scope: What can the generated code see and modify? If it has access to the full environment, how do you reason about what it might do?
  • Persistence: Does generated code persist across LLM calls? Can one generation build on what a previous generation defined? If so, the program's behaviour becomes path-dependent and very hard to analyse.
  • Types: If the host language has a type system, can the generated code be type-checked? This is non-trivial — the code is a string until it's parsed and compiled, and the types it needs to satisfy depend on the runtime context.
  • Verification: Can you make any formal guarantees about a program that includes LLM-generated code? The generated code is, from the program's perspective, adversarial input that happens to be executable. How do you bound its behaviour?

This connects to questions I've been thinking about in the context of policy-as-code and runtime safety — the idea that you need a formal layer that can reason about and constrain the actions of an agent, whether those actions come from tool calls or generated code. If the LLM can generate arbitrary code in the host language, the "tool schema as safety boundary" approach breaks down entirely, and you need something more expressive.

Where Does This Leave Us?

I think "agentic programming" is a genuine new programming paradigm — or at least a genuinely new set of patterns — and it deserves to be studied with the same rigour we apply to concurrent programming, reactive programming, or any other paradigm that introduces a new source of nondeterminism into program execution.

Some questions I think are worth pursuing:

Typing and protocols. Can we design type systems that capture the interaction protocol between a program and an LLM? Something like session types, but for agent loops — specifying what tool calls are valid at each stage, what the expected response shapes are, and how the conversation state evolves.

Effect systems. LLM calls are effects — they're nondeterministic, they have latency, they cost money, they can fail. Can we use effect systems to track and manage these, the way we track IO or exceptions?

Gradual sandboxing. Rather than a binary choice between "full sandbox" and "same environment," can we design capability-based systems that give generated code precisely the access it needs? This sounds a lot like object-capability security, which has been studied extensively but never achieved mainstream adoption. Maybe LLMs provide the motivation.

Formal methods for agentic programs. Can we verify properties of agentic programs despite the nondeterminism of the LLM? Perhaps by treating the LLM as an oracle with specified contracts (preconditions, postconditions), and verifying the rest of the program relative to those contracts.

Self-modifying programs done right. The history of self-modifying code is mostly a cautionary tale. But LLMs give us a new form of it — one where the modifications are guided by natural language intent and constrained by the available API surface. Can we learn from past mistakes and design safe, useful forms of LLM-driven metaprogramming?

We're at the beginning of figuring this out. The current state of agentic programming is roughly where concurrent programming was before we had good abstractions — lots of ad hoc patterns, lots of things that work until they don't, and a pressing need for foundational thinking. I suspect the PL community has a lot to contribute here, if it chooses to engage.
